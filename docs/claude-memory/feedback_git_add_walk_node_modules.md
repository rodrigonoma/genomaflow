# `git add -A` em working tree com `node_modules` causa timeout

**Regra:** se um container faz `git add` programaticamente em working
tree que tem node_modules (qualquer scope), **NUNCA** usar `git add -A`
ou `git add .` — sempre allowlist explícito:

```js
await run(['add', '--',
  'apps/api/src', 'apps/api/tests',
  'apps/worker/src', 'apps/worker/tests',
  'apps/web/src',
  'docs',
]);
```

**Why:** `git add -A` walka o working tree inteiro pra detectar
untracked + modified. Mesmo arquivos em `.gitignore` (node_modules) são
STAT'ed durante o walk pra decidir se devem ser ignorados. Em monorepo
JS com `npm ci` baked em múltiplos apps, são milhões de arquivos.

Sem timeout: `git add -A` pode demorar 5-15min, ou pendurar.
Com timeout: explode com `git add TIMEOUT after Ns` e usuário fica
sem clue do que aconteceu.

**Incidente real (Trello QA Agent, 2026-05-14):**
Worker container `/app/repo` tinha `apps/api/node_modules` e
`apps/worker/node_modules` (baked via `npm ci` no Dockerfile pra fazer
`run_tests` funcionar). `git add -A` no fluxo de fix do agente
timeoutou em 60s. Sintoma no card: `🚨 Agente falhou: git add TIMEOUT
after 60s`.

**Como aplicar:**

1. Identificar paths editáveis (se o código tem `EDITABLE_PREFIXES` ou
   similar, usar como fonte da verdade):
   ```js
   const EDITABLE_PREFIXES = [
     'apps/api/src/', 'apps/worker/src/', 'apps/web/src/',
     'docs/', 'apps/api/tests/', 'apps/worker/tests/',
   ];
   ```

2. Passar como argumentos a `git add -- <paths>` (com `--` pra evitar
   ambiguidade de glob):
   ```bash
   git add -- apps/api/src apps/worker/src apps/web/src docs apps/api/tests apps/worker/tests
   ```

3. Se um path não existe (ex: agente não tocou tests), git pula
   silenciosamente — não é erro.

**Pacotes/cenários afetados:**
- Containers que rodam `git add` em working tree monorepo
- CI runners que clonam o repo + fazem npm install antes de git add
- Qualquer fluxo `git add -A` em diretório com node_modules

## Não regredir

❌ Não usar `git add -A` em `/app/repo` ou similar working tree com node_modules
❌ Não confiar no `.gitignore` pra evitar walk — git STAT-a mesmo arquivos ignorados
❌ Não bumpar timeout de 60s pra 600s como workaround — sintoma volta

✅ Sempre allowlist explícito quando container faz `git add` programático
✅ Sempre `--` antes dos paths (defesa contra path que parece flag)
✅ Sempre timeout no spawn de operações git em background (60s seguro pra ops normais)
